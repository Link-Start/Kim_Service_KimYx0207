# 大师蒸馏协议

大师蒸馏不是模仿某个人的口吻。

它只提取可迁移的判断结构、取舍规则、验收标准和失败边界。

## 触发条件

当用户要求“像某人一样判断”“学习某套方法”“提炼高手做法”“总结风格背后的方法”时加载。

可见输出里可以提到具体人物、作品、案例或来源，但只能作为证据或样本。内部方法必须转成抽象模型。

## 蒸馏对象

只蒸馏四类内容：

- 判断顺序：先看什么，再看什么。
- 取舍规则：什么时候推进，什么时候放弃。
- 证据偏好：什么信号更可信，什么信号只是噪音。
- 输出标准：最后交付什么形态，怎样才算可用。

不蒸馏：

- 口头禅。
- 人设。
- 情绪姿态。
- 神秘化表达。
- 无法验证的个人传奇。

## 步骤

1. 收样本：列出用户提供或可验证的样本，不够就标注缺口。
2. 抽动作：把样本里的动作改写成“主体 -> 输入 -> 判断 -> 动作 -> 输出”。
3. 抽标准：找出通过标准、停止标准和不做什么。
4. 去人格化：删掉具体人物身份，只保留抽象规则。
5. 回到任务：把规则应用到用户当前问题，生成可执行结果。

## 输出形态

```markdown
可迁移内核：
[一句话说明这套方法真正解决什么问题。]

抽象规则：
- 先看……
- 如果……就……
- 不用……作为判断依据。

用于当前任务：
[给出一条具体执行路径、测试假设、通过标准、停止标准和硬缺口。]
```

## 硬缺口

如果样本不足，必须直说：

- 缺哪些样本。
- 缺口会影响哪个判断。
- 先用什么最小样本补齐。

不要把少量印象包装成完整方法。

## 验收

- [ ] 没有把具体人物写成内部角色。
- [ ] 具体姓名只作为样本或证据出现。
- [ ] 至少抽出一个判断顺序、一个取舍规则、一个证据偏好。
- [ ] 已说明测试假设。
- [ ] 已说明硬缺口。
- [ ] 输出能直接用于当前任务。
