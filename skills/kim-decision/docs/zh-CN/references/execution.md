# 执行计划协议

执行计划不是把建议拆成待办事项。

它要把判断变成可以开始、可以验收、可以停止的动作链。

## 触发条件

当用户要方案、计划、落地、执行、推进、MVP、交付、增长、变现、验证时加载。

## 核心问题

- 谁执行？
- 拿什么输入开始？
- 第一个动作是什么？
- 产出物长什么样？
- 什么信号说明继续？
- 什么信号说明停止？
- 哪个假设必须被测试？
- 哪个缺口是硬缺口？

## 输出标准

执行计划必须包含：

- 目标：这轮执行要改变什么。
- 执行者：谁做，不写成抽象组织。
- 输入：需要用到的材料、数据、样本、客户、渠道或文件。
- 动作：第一步要能在 24 小时内开始。
- 产出：可检查的东西，不是“提升”“优化”“完善”。
- 测试假设：这一步实际在验证什么。
- 通过标准：继续投入的阈值。
- 停止标准：停止、改方向或回收范围的阈值。
- 硬缺口：没有它就不能做判断的信息。
- 不做什么：防止范围膨胀。

## 自然格式

不要默认输出表格。

先给判断，再给为什么这条路赢，然后给动作链。只有在用户要审计、验收或多人协作时，才把执行计划写成表格。

好的自然格式：

```markdown
先跑一条最窄路径：让一个真实用户在 24 小时内完成一次可见动作。原因是现在最大的风险不是功能不够，而是不知道用户是否愿意开始。

今天做三件事：
1. 找 5 个目标用户，给他们同一个入口。
2. 记录他们从打开到完成第一步的卡点。
3. 只修阻止第一步发生的问题。

测试假设：用户看见入口后知道下一步做什么。

通过标准：5 人里至少 3 人能在无解释情况下完成第一步。

停止标准：5 人里少于 2 人理解入口，先重写定位和入口文案，不继续加功能。

硬缺口：当前没有真实用户的首次使用记录。
```

## 禁止动作

- 不用“优化体验”“加强转化”“完善闭环”代替动作。
- 不把多个方向并列成菜单，除非用户明确要选项。
- 不把测试假设藏在通过标准里。
- 不跳过停止标准。
- 不把硬缺口写成普通风险。

## 验收

- [ ] 第一动作能在 24 小时内开始。
- [ ] 每一步都有执行者、输入、产出。
- [ ] 至少一个测试假设被明确写出。
- [ ] 通过标准和停止标准都是可观察的。
- [ ] 硬缺口被单独标注。
- [ ] 有明确的不做什么。
