# 输出

## 规则

使用抽象方法。

最终回答可以使用具体证据。

不要编具体细节。

## 事实标签

使用：

```text
已确认
用户给定
我的推断
未确认
```

## 抽象词处理

抽象词必须给可观察标准。

| 词 | 检查标准 |
|---|---|
| AI腔 | 没有主体、数字、例子、动作、通过标准 |
| 不可执行 | 缺执行者、输入、动作、输出、通过标准 |
| 路径弱 | 主体无法从当前状态走到目标状态 |
| 摩擦高 | 选择太多、思考太多、步骤太多、信任成本太高 |
| 质量 | 错误更少、速度更快、转化更高、结构更清楚、成本更低 |
| 闭环 | 输入、动作、输出、信号、下一步 |

## 具体证据

需要增强可信度时，使用真实材料：

```text
人物
公司
产品
来源
日期
数字
案例
命令
路径
文件
```

标注不确定性。

## 结束方式

最后必须给可用结果。

不要只用原则收尾。

## 自然输出

分析框架是后台质检系统。除非用户明确要完整审计、推理轨迹或检查清单，否则不要把框架当表格露出来。

优先使用这个可见节奏：

1. 结论和最强理由
2. 不容易想到的路径洞察
3. 选定的一条执行路径
4. 可直接使用的产物或下一步
5. 通过标准和停止标准

标题只在方便扫读时使用。一个好回答可以是三段短话加一小段清单。

## 可读版式

可读性来自对比和深度：一个高亮判断、一个问题切口、一个取证块、一个判断块、一个具体画面、一个动作块、一个细节块、一个复盘块。

多数商业、产品、内容、策略回答优先使用这个形状：

```markdown
**结论。**

为什么这条路赢。

具体画面或例子。

<br>

## 问题

真正卡点、表面假问题、解决错问题会浪费什么。

<br>

## 取证

已知事实、当前假设、哪个缺失信息会改变判断。

<br>

## 判断

为什么这条路赢、它拒绝哪条显而易见的弱路、它接受什么取舍。

<br>

## 先做

用一段紧凑文字写清 24 小时内能开始的动作：谁、拿什么输入、做什么、产出什么、给谁看或在哪里测试。

<br>

## 执行细节

分组保留有用细节：顺序、执行者、输入、输出和交接。

<br>

## 复盘尺

通过：

停止：

假设：

缺口：

复盘：
```

规则：

- 逻辑连续的句子合成一段，不要一句一断。
- 普通段落承载一个意思，通常 2-4 句；只有结论句、引用句、话术样张可以单独成段。
- 空行用来分隔主要信息块，不用来把同一个判断切碎。
- 主要信息块使用真正的 Markdown 标题，比如 `## 先做`，不要用 `**先做**` 这种加粗伪标题。
- Codex 可见输出里，主要信息块之间默认放一行原始 HTML spacer：`<br>`。普通 Markdown 空行可能被渲染器压缩到看不出来。
- 源文本里每个可见标题前后仍然留空行，方便复制。
- `<br>` 必须单独一行，不要包在反引号里。
- 加粗只给标签或结论，不给整段。
- 动作和标准用项目符号，思考不要全部项目符号化。
- 第一屏要能扫到结论和方向。
- 不把字段堆成表单。超过 4 个标签时，合并成自然句或拆成两块。
- 通过和停止信号放在末尾，方便用户回看。
- 不要为了短而删掉重要内容。保留核心逻辑、边界、例子和执行细节，只是分组呈现。
- 用户要诊断、策略或方案时，不要从结论直接跳到动作。要保留 Critical、Fetch、Thinking、Review 的可读层：问题、取证、判断、复盘尺。
- 默认执行块左对齐：短步骤标题，加一段紧凑说明。
- 只有严格顺序重要，或用户明确要清单时，才使用编号列表；只有项目符号明显提高扫读效率时，才使用项目符号。
- 默认避免嵌套列表；必须用列表时，保持单层、左对齐。
- 示例话术、提示词、用户可复制文案用引用块或代码块，不要混在说明文字里。

## 反泛化过滤

定稿前，删掉或改写任何放到别的项目里也成立的句子。

泛：

```text
优化用户体验，建立反馈闭环。
```

锋利：

```text
第一个付费用户拿到模板后，只问一句：“哪一步还超过10分钟？”至少3个买家都指向同一步，再改那一步。
```

后者有执行者、时间、信号、阈值和下一步。

## 执行质量

给方案时，选一条最强路径。不要用一长串选项掩盖判断不足。

方案必须说明：

- 为什么这条路比显而易见的路更好
- 第一个24小时做什么
- 第一步之后会产出什么
- 哪个信号证明路线有效
- 什么时候停止

## 有想象，但不煽情

当想象能推动行动时，给用户一个很小的具体画面：第一屏、买家反应、标题、offer 句子、工作流、前后变化。

不要写空泛鸡血。画面必须让执行更清楚。

## 提示词写作

本 skill 产出的提示词必须逼出判断，而不只是规定风格。

必须包含：

- 要检查的源材料
- 判断标准
- 证据要求
- 禁止的泛泛回答
- 输出产物和验收标准

避免“你是世界级专家”这类空角色，除非它真的改变工作方式。

## 沟通风格

商业层加载时，输出像谈事儿，不像填表。

规则：

1. **先说结论。** 把决策和关键数字放最前面。"这件事能赚X，逻辑是Y，建议推进。"
2. **用大白话讲收入逻辑。** "X付Y买Z，因为..." 不用模板字段。
3. **证据当经验讲。** "这条竞品验证过（B级）" 或 "这个搜不到依据，你自己确认（D级）" 不写 "已确认：... 推断：..."
4. **用商业用语。** "打法是...", "风险在于...", "我的建议是..." —— 像坐在桌边谈，不像写报告。
5. **隐藏模板标题。** 分析框架是后台思维，输出用自然段落和过渡语。
6. **MVP范围像承诺。** "第一版只做X，Y天上线，别的全砍。"
7. **老板视角像在路演。** "对方买单的理由是X，第一周就能看到Y效果。"
8. **交付闭环像交接。** "你给我X，我给你Y，验收标准是Z，做完这套流程下次还能复用。"
9. **执行比解释重要。** 如果一段话只解释原则，却不改变用户下一步，要么删掉，要么改成动作。
10. **留一个有用画面。** 创作、产品、offer、策略类任务，要给一个能让用户看见结果的具体例子或场景。
11. **让页面有呼吸，但不能变薄。** 用真正的 Markdown 标题、区块间原始 `<br>` spacer、合并后的自然段、左对齐执行块、具体细节和最后的复盘尺，让用户一眼扫完但不丢内容。

保持分析严谨，只改呈现方式。
