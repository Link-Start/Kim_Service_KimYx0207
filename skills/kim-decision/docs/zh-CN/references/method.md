# 方法

## 框架

```text
意图 -> 主体 -> 路径 -> 约束 -> 证据 -> 最小验证 -> 模型 -> 硬门 -> 输出
```

## 意图

写真实变化。

差的写法：

```text
提升质量。
```

好的写法：

```text
让接收者在 3 分钟内理解下一步，并完成一个可验证动作。
```

## 主体

写谁会经历这个结果。

不要固定角色。

根据任务生成主体。

例子：

```text
必须做决定的人
接收草稿的人
运行流程的人
进入产品的人
承担风险的人
```

## 路径

写移动过程：

```text
当前状态 -> 动机 -> 理解 -> 行动 -> 阻力 -> 信号 -> 状态变化 -> 延续动作
```

没有路径，不写完整方案。

## 约束

写硬限制。

例子：

```text
7 天
1 个人
不开发
只能用公开数据
必须可回滚
不能增加现金流风险
```

## 证据

使用四个标签。

### 已确认

来自真实资料、工具、文件、官方信息、可核验证据。

### 用户给定

用户提供，但没有独立验证。

### 我的推断

基于已有信息推导。

### 未确认

证据不足，不能当事实。

### 证据等级

给每条证据打等级：

- A：真实数据、真实客户、真实收入、已验证的结果
- B：公开案例、竞品验证、已发布的基准数据
- C：基于已有信息的合理推理，未验证
- D：猜测——不能作为决策依据

如果关键决策只依赖 C 或 D 级证据，标记为数据缺口。

关键决策依赖 C/D 级证据时，先尝试用可用工具验证（联网搜索、文件读取、API查询）。验证成功则升级等级并标注来源，验证失败则标记"无法验证，需用户确认"。关键决策不能只靠 D 级证据。

## 最小验证

最小验证必须能改变判断。

字段：

```text
目标
输入
动作
输出
通过标准
失败信号
下一步
不做什么
```

## 模型

模型不是装饰。

每个模型必须产出：

```text
发现
修正
```

## 硬门

硬门用于阻止跳步骤。

## 输出

最后给可用结果。

可用结果是能复制、运行、发送、决策、检查的东西。
