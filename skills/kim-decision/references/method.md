# Method

## Frame

```text
Intent -> Subject -> Path -> Constraint -> Evidence -> Minimum Test -> Models -> Gates -> Output
```

## Intent

Write the real change.

Bad intent:

```text
Improve the plan.
```

Good intent:

```text
Make the receiver take one measurable next action.
```

## Subject

Write who experiences the result.

Do not use a fixed role list.

Use the role created by the task.

Examples:

```text
the buyer
the reader
the operator
the reviewer
the system user
the decision owner
```

## Path

Write movement.

```text
current state -> motive -> interpretation -> action -> resistance -> signal -> state change -> continuation
```

A plan without a path is not ready.

## Constraint

Write the hard limits.

Examples:

```text
7 days
one person
no engineering work
public data only
must be reversible
must not increase cash-flow risk
```

## Evidence

Use four labels.

### Confirmed

Use this for checked facts.

### User-provided

Use this for information from the user that has not been independently checked.

### Inference

Use this for conclusions drawn from available facts.

### Unconfirmed

Use this for missing evidence.

### Evidence tiers

Tag each evidence item:

- A: real data, real customers, real revenue, verified outcomes
- B: public case studies, competitor validation, published benchmarks
- C: reasonable reasoning from available facts, unverified
- D: guesswork — do not use as decision basis

If a key decision rests on C or D evidence only, state it as a data gap.

## Minimum Test

The minimum test must be small enough to run before a larger build.

Required fields:

```text
goal
input
action
output
pass condition
fail signal
next step
do not do
```

## Models

Use models to improve judgment.

Do not use model names as decoration.

Each model must produce:

```text
finding
fix
```

## Gates

Use gates to stop skipped reasoning.

## Output

End with a usable artifact.

A usable artifact is something the user can copy, run, send, decide with, or check.
