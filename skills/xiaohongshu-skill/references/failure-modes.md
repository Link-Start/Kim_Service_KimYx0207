# 失败模式与拦截动作

| 失败模式 | 典型表现 | 为什么差 | 拦截动作 |
| --- | --- | --- | --- |
| 模板爆款味 | 标题、emoji、标签堆满，但没有真实场景 | 免费 prompt 已经很多，用户不会为此买单 | 先填真人细节补位表，再写标题 |
| 标题兑现失败 | 标题说“3 步解决”，正文只有情绪 | 读者点进来失望，信任下降 | 用标题封面兑现矩阵检查 |
| 伪经验 | “我试过后涨粉很多”但用户没给证据 | 可能虚假营销，也容易被用户识破 | 改成“我自己的流程变化”，标待补证据 |
| 硬营销 | 正文突然引导私信、报名、购买 | 像广告，破坏分享感 | 先给真实价值，再克制承接 |
| 明面导流/导私 | 评论或私信承接写“私信我、发我、想了解课程、领取资料、加群、报名、咨询课程”等 | 平台风控和封号风险高，也会把真实分享拉成硬广 | 改成公开区低风险回应和站内互动边界；不主动导私、不索要联系方式；主动问起时只给公开可核验信息和适用边界 |
| 假真实感 | 画面太干净、太像图库或商业海报 | 用户会觉得是广告或 AI 占位图，不像真实账号 | 要求具体物件、生活痕迹、手机拍摄感和不完美细节 |
| 批量同质 | 同一个结构换主题套 50 篇 | 低质内容，无法形成账号信任 | 要求每篇至少 4 类真实信号 |
| 过度焦虑 | 制造不做就落后的恐吓 | 低质营销感 | 改成具体问题和适用边界 |
| 视觉断裂 | 图文页和正文不是一件事 | 做图落地困难 | 每页写画面建议和素材需求 |
| 兜底抢主路径 | Image2 可用时仍先生成 SVG/发布模拟包 | 用户要的是照片级/真实感图片，SVG 只是结构兜底 | 按 Image2 -> 其他图片生成 MCP/工具 -> SVG 兜底顺序执行；Image2 全套成功时停止生成 SVG |
| 决策点不足 | 模糊意图后直接写标题、正文和全套图 | 模型会把未确认信息当事实，容易瞎编 | 先输出策略补全与待确认决策，再给标题/封面选项 |
| 封面未验证就批量出图 | 还没确认封面风格、标题、小图可读，就生成 6-8 张内页 | 错误风格会被放大，返工成本高 | 先生成 1 张带已确认标题/封面字的封面图，信息流小图通过后再批量生产 |
| 封面没有标题 | 视觉图只有氛围或桌面，没有选中标题/封面字 | 小红书封面承担点击和主题判断，没标题无法验收 | Image2 首选一体化生成带标题封面；文字失败时才用图片层 + 可编辑文字叠加 fallback |
| 场景硬编码串场 | 某个赛道内容的信息流测试出现另一个赛道的旧案例文案 | 说明脚本或模板没有按输入域重算，交付不可信 | 脚本参数化账号、同类标题和赛道；review 检查是否残留旧 domain |
| 轻量草稿抢路由 | “不想像广告/不要 AI 味/真实图文”请求被旧轻量草稿路线直接输出 7 段草稿 | 完整 workbench 的策略补全、视觉验证、批量图链路和选项合同全部丢失 | 这些信号必须转用当前唯一 `xiaohongshu-skill`；回归测试用轻量输出反打 validator |
| 选项被压缩 | 用户已选标题或封面字后，输出只剩最终选择，备选标题、封面字、视觉方案消失 | 用户无法比较和回退，也看不到模型如何做取舍 | 标题保留至少 8 个、封面字保留 3 组、视觉方案保留 2-3 个；已选项只加“已确认/已选” |
| 标题多选不足 | 标题区只有 1 个最终标题或 6 个同质标题，没有多选/备选路线 | 小红书标题需要对角度、语气、承诺强度做选择；单标题会把模型押注伪装成用户确认 | 原生选择阶段确认 3 个标题方向/标题组；最终标题区至少保留 8 个可发布标题，并标出已选、推荐和备选 |
| 正文路线与营销目标不匹配 | 搜索教程被强塞故事转折，比较内容没有判断标准，或信任种草只有空观点没有过程证据 | 读者得到的内容与标题承诺、需求阶段和行动目标不一致 | 按营销决策重选路线：信任/种草写真实场景、过程证据和克制边界；搜索/收藏写方法、步骤、易错点和可保存结论；比较/咨询写标准、证据和适用边界；观点/讨论写判断、例子或反例和讨论边界 |
| 哲学性结尾空泛 | 结尾只写“开始行动吧/希望对你有帮助/学习要落地”等泛话 | 没有把本条经历上升到学习、表达、行动的本质，像 AI 生成结语 | 结尾要具体落到“把想法拆成能运行的动作”“不是收藏答案，而是让一个东西动起来”等可感知表达 |
| 排版合同漂移 | 输出段名变成“策略判断/标题备选/图片提示词/发布前避坑”，15 个工作台段落缺失或换序 | 下游验证、出图、封面确认和批量生产无法接上 | validator 强制 15 个 `##` 段落按固定顺序出现，禁止轻量草稿段名替代 |
| 标题正文远离图片 | 标题和正文在前面，图片结果在后面，中间隔着大量过程说明 | 用户发小红书时需要对照图片和发布文案，分散会造成漏图、漏文案和重复返工 | 最终 `标题` 与 `正文草稿` 放在 `批量图片生产` 后面，和图片清单一起交付 |
| 普通聊天卡冒充弹窗 | 只在正文里列“你选 A/B/C”，没有 Codex `request_user_input` 或 Claude Code `AskUserQuestion` 证据，却继续执行 | 用户要的是一步步确认；没有原生确认就继续跑，会把模型假设当用户选择 | 原生确认工具不可用时写 `阻塞：缺少原生确认工具` 并停止；用户同轮已明确选择时写“用户已明确确认” |
| 验收 prompt 污染 | 新窗口测试时在用户模糊意图后追加“先决策、优先 Image2、不要明面导流、不要编造”等答案 | 这会让测试变成提示词作弊，无法证明 skill 自己能识别流程、工具优先级和风险边界 | 新窗口验收只发送用户给出的原始模糊意图；额外规则必须写进 skill 本体、validator 或 eval，不得塞进测试 prompt |
| 封面后文字确认 | 生成封面后，最后写“下一步确认封面，通过我再批量生成”，但没有弹框 | 大段文字里用户看不见关键决策，批量生产被藏在结尾，实际体验像没给选择 | 封面小图测试后立即调用 `request_user_input` / `AskUserQuestion`，选项是“通过并批量生成 / 先停查看 / 不通过改 Brief”；无工具则阻塞 |
| 已确认后降级成文案 | 用户已经要求图文/生成/引流贴，并完成选题和视觉决策，弹框仍推荐“文案+脚本，不生成图片” | 把用户的完整生成目标倒退成半成品，等于让用户重复决策并放弃图片 | 前置决策完成后直接生成封面候选；只有用户明确说先不要出图时，才推荐文案+脚本 |
| HookPrompt 改写降级 | 原始输入只说用小红书 SKILL 生成引流贴，但优化提示词 scope 写“不需要制作图片”，后续 skill 按纯文案执行 | 上游改写越权，覆盖了用户真正目标，导致图片能力消失 | 原始输入优先；没有用户原话确认“不出图”时，忽略优化提示词里的纯文案限制 |
| 一次性长文替代工作台 | 策略、标题、正文、封面、批量图一次性输出，没有阶段停顿 | 看起来内容很多，但关键决策没有被用户控制，后续返工成本高 | 每阶段只输出当前阶段，确认前不得跨到下一阶段；最终 15 段只在前置确认完成后输出 |
| 提示词冒充出图 | 只给 Image2/GPT Image 提示词，没有图片文件或可见生成结果，却写“已生成封面” | 用户要真实视觉图，不是另一个 prompt；也会误导验收 | `封面视觉验证` 必须写图片文件/宿主可见结果/`阻塞：缺实图证据` 三选一 |
| 旧图冒充新图 | 新窗口测试或本轮生成请求里，直接拿历史 `outputs/` 目录、旧封面、旧内页或示例文件交付 | 用户要验证当前 skill 能不能决策、出图和批量生产；复用旧包会掩盖流程失效 | 旧产物只能标为历史参考；除非用户明确要求复用，否则必须重新走原生选择和本轮图片生成证据 |
| 缺封面确认就批量生产 | 生成了封面提示词或封面图，但未做手机信息流小图测试、也未获用户确认，就继续做 6-8 页 | 一旦封面方向错，整套内页都会错 | 封面图、小图测试、用户确认三项都通过后才批量；否则 `交付状态：partial/blocked` |
| 无图仍称交付级 | 最终只有正文、提示词、SVG 或结构包，却说完整交付 | 混淆结构预览和照片级最终图，直接伤可信度 | 没有 image2/图像模型直出图或已有图片素材时，不得写 `交付状态：交付级` |
| 小字贴纸补丁 | 主标题小，另加右下角小贴纸、小角标、小标签写“真实学习桌面/过程记录”等 | 手机信息流里看不清，像掩耳盗铃，不是真实感 | 删除无意义小贴纸；把已确认封面字放大为主标题，缩略图先读标题 |
| Image2 被降级成照片层 | Image2 可用时，只让它生成无字桌面照片，再人工叠大字 | 违背 Image2 优先，整体文字风格和图片关系会割裂 | 首先用 Image2/GPT Image 一体化生成完整封面；只有错字、不可读、排版失败或用户要求可编辑时才进入 overlay fallback |
| overlay 冒充一体化生成 | 输出写“Image2 已生成封面”，但实际是 photo layer + editable overlay | 用户会误以为 Image2 审美和文字都通过，验收层级被混淆 | 输出必须标明 `Image2 integrated cover` 或 `Image2 photo layer + editable overlay fallback`，fallback 必须说明原因 |
| 贴顶横幅导致下方空 | 标题块贴着画面最上方，下面大面积桌面/墙面没有信息重心 | 字虽然大，但画面像临时盖了个横幅，不像真实小红书封面 | 标题块下移到上中部或左中部，与电脑、笔记、桌面主体形成整体重心 |
| 散文式封面 prompt | 只写“真实学习桌面、自然光、标题融入画面”等一段话，没有视觉任务、构图、文字层级和验收标准 | Image2 会自由发挥，容易出现贴顶横幅、小字补丁、画面空、标题和主体断裂 | 封面提示词前强制输出 `Image2 一体化封面 Brief`，字段包含一秒先读、构图、镜头光线、文字层级、验收标准和 fallback 触发 |
| 默认假设冒充人类决策 | 用户没确认视觉路线或 Brief，输出只写“推荐方案”，却继续调用 Image2 出图 | 这就是抽卡，会把模型偏好当用户审美，返工概率高 | 关键节点必须写 `human_confirmed` / `explicit_auto_permission` / `blocked_pending_human_decision`；默认假设只能出方案，不能出图 |
| 不满意后继续抽卡 | 封面被指出不好看后，直接重新生成多张，没有改模板路由、构图或 Brief | 只是换随机结果，不是解决审美决策问题 | 先写失败类型、修正后的 Brief 差异和下一步确认状态；每个已确认 Brief 只生成 1 张主候选 |
| 完整交付只给单图 | 用户要求图片也做出来或完整图文交付，但最终只显示 1 张封面，并说等用户确认后再做内页 | 这把过程检查当成最终结果，用户实际看不到整套图，无法发布或验收 | 封面通过小图测试后必须弹框；用户选择批量生成后必须继续生成封面 + 6-8 页内页图；只生成 1 张必须判为 `partial/blocked` |
| 用户确认后还问范围 | 用户已经完成方向/标题/视觉选择，或已经选择“通过并批量生成”，系统又弹“这次要做到哪一步” | 这是把用户决策清零，逼用户在完整交付和半成品之间重复选择 | 前置选择完成后直接生成封面；批量确认完成后直接生成所有内页，只在工具失败时询问重试或降级 |
| 真实素材停图 | “真实女生留学生活/真人写真/真实交友”等请求最终写“最好用真实授权素材，本轮没有生成最终图片” | 用户要的是生成真实感图片，缺实拍只应影响声明和替换建议，不应取消 Image2/宿主原生生图 | 直接进入 Image2/宿主原生图像，标注生成素材和可替换实拍；只有宿主拒绝、工具失败或用户要求不出图才 blocked |
| 文字信息卡绕过 Image2 | 老金课程、知识卡、图片文字为主体的请求直接用 PowerShell/HTML/SVG/本地 PNG 排版，没先调用 Image2/宿主原生图像 | 文字为主体不等于不用 Image2；用户在测的是图片能力和一体化图文路线 | 先用 Image2/宿主原生图像做一体化文字封面；错字、不可读、排版失败或用户要求可编辑时，才写明 fallback 并本地排版 |
| 文件交付无聊天成稿 | 最终只给 `note.md`、图片目录、文件链接和 DONE，聊天里没有标题、正文、评论私信、标签 | 用户必须打开文件才知道交付了什么，主聊天体验没有结果感，也无法即时判断内容质量 | 最终回答必须有聊天内可见成稿：图片摘要、已确认标题、至少 8 个标题备选、完整正文、评论私信、标签和素材缺口；文件路径只作为补充归档 |
| 工作台倾倒成最终回复 | 最终聊天直接输出 `策略补全与决策提醒` 到 `最终成品自检` 15 段，正文和图片被埋在大量过程信息里 | 用户要的是可验收发布包，不是调试报告；这会造成“信息很多但结果很弱”的体验 | 完整 15 段只写归档 Markdown；聊天最终按满意交付形态输出：完成摘要、封面预览、标题、正文、承接、标签、图片清单 |
| MCP 抢 Image2 | 新窗口里直接调用 MiniMax、豆包、即梦、Midjourney 或其他图片 MCP，但没有先证明 Image2 不可用或失败 | 违背“Image2 最高优先级”，也会把 MCP 当成主路径而不是降级路径 | 先写 `Image2 优先级证据`：Image2 已调用并失败 / 当前宿主无 Image2 可调用工具 / 用户明确要求不用 Image2；没有证据就不得调用 MCP |

## 兜底策略

如果素材太少：
- 输出“低素材版本”，只写观点短帖或待补清单。
- 不写案例、不写效果、不写教程承诺。
- 明确告诉用户补什么素材后才能升级成完整笔记。

如果用户只想要很短：
- 只输出标题、封面字、正文短版和最终成品自检。
- 仍保留真实素材缺口。

如果用户要求爆款承诺：
- 拒绝承诺结果。
- 改成“提高可读性和兑现度”的目标。
