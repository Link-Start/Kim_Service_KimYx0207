# 验收方法

## 什么时候读

当需要判断一条笔记、图片包或技能运行是否合格时读取本文件。

## 触发验收

检查用户需求是否真的属于小红书图文：

- 应触发：完整小红书图文笔记、明确包含封面和内页的发布包、整套图文视觉制作。
- 不应触发：一句标题、标题组、封面字、正文润色、评论回复、标签、单个图片提示词、普通长文润色、PPT、邮件、论文、代码。
- 模糊场景：先判断是否真的要完整发布包；没有完整封面与内页需求时不要强行走本技能。

## 输出验收

完整工作台文件必须有 15 段结构：

1. 策略补全与决策提醒
2. 任务判断
3. 真人细节补位
4. 选题角度
5. 封面字
6. 图文页脚本
7. 视觉类型判断
8. 图片生成与配图提示词
9. 封面视觉验证
10. 批量图片生产
11. 标题
12. 正文草稿
13. 评论与私信承接
14. 标签建议
15. 最终成品自检

部分任务可以少段，但必须说明本次不做哪些内容。聊天最终答复另按发布版交付验收，不要求也不允许默认倾倒 15 段工作台。

## 聊天交付验收

完整图文完成后，最终聊天必须像发布包，而不是调试报告。检查是否包含：

- 完成摘要：原生决策、封面生成、批量生成和校验状态。
- 封面预览或本轮新生成封面路径。
- 发布标题首选和至少 8 个标题备选。
- 可复制正文草稿。
- 承接方式和平台边界。
- 3-6 个标签。
- 本轮新生成图片清单和完整工作台文件路径。

聊天最终答复不应整段输出 15 个 `##` 工作台标题。若用户明确要求完整工作台，则可以展示；否则完整工作台放文件。

## 质量验收

逐项检查：

- 标题至少 8 个。
- 封面字至少 3 组。
- 正文与营销路线一致：信任/种草有真实场景、过程证据和克制边界；搜索/收藏有方法、步骤、易错点和可保存结论；比较/咨询有判断标准、证据和适用边界；观点/讨论有明确判断、例子或反例和讨论边界。
- 评论与私信边界不主动导流。
- 图片路线写清 Image2、降级或阻塞原因。
- 缺实图时不写成交付级。
- 本轮图片证据必须是 `imageGeneration`、`image_gen`、本轮新生成路径或宿主可见结果；历史 `outputs/` 只能算旧参考。
- 最终聊天信息量要服务发布验收，不能用长工作台掩盖缺图、缺决策或缺正文重点。
- 完整包必须有机会研究、营销决策和逐页视觉导演表；每页都要能回答“画什么、字在哪、为什么这样画”。
- 同一主题换成不同内容目标（停留、收藏、信任、咨询）时，标题、正文路线、封面版式或内页角色至少有三项明显变化。
- 若用户原始意图是图文/生成/引流贴/完整发布包，前置方向确认后不得再推荐“文案+脚本，不生成图片”；触发本 skill 后没有纯文案路线，用户要求暂停时只能停在当前流程状态。
- 若用户已经完成前置方向选择或已选择“通过并批量生成”，不得再出现“这次要做到哪一步 / 是否只要文案 / 文案+脚本推荐”类范围降级弹框。

## 脚本验收

有成稿文件时运行：

```bash
node scripts/check-rednote-note.mjs <output.md>
node scripts/check-page-direction-fixtures.mjs
node scripts/check-narrative-route-fixtures.mjs
```

需要生成结构预览或交接包时，再按实际输入运行：

```bash
node scripts/build-rednote-visual-pack.mjs <output.md> <output-dir>
node scripts/build-rednote-realshot-handoff.mjs <output.md> <output-dir>
node scripts/build-rednote-final-replacement-pack.mjs <output.md> <output-dir>
```

脚本通过只证明结构和规则检查通过，不等于真实图片已经发布可用。

## 复查结论

复查时只给三类结论：

- `通过`：结构、内容、图片证据或阻塞说明都完整。
- `部分通过`：文案可用，但图片、真实素材或用户确认不足。
- `不通过`：缺关键段落、导流违规、编造事实、图片证据造假或标题正文不兑现。
