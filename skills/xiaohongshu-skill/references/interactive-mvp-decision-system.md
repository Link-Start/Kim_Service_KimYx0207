# 交互式弹框与封面 MVP 决策系统

## 什么时候读

当用户只给出模糊主题、要求“生成小红书/图文笔记/引流贴/种草贴/封面内页/完整发布包”、需要通过 Codex 或 Claude Code 的交互式弹框补全意图，或用户已经确认封面后要批量生成整套图文时读取本文件。

## 目标

小红书 skill 的产品体验应是：用户给一个模糊意图，agent 通过少量原生弹框补全关键决策；封面图和封面文字这个 MVP 被确认后，立刻并行生成完整发布包，包括至少 8 个标题、1 篇正文、封面图和 6-8 张内页图。

不要把交互做成问卷。只问会改变成品方向的选择；其他细节由 AI 默认补全，并在工作台里记录默认依据。

## 原生弹框优先级

优先级从高到低：

1. Codex `request_user_input`。
2. Claude Code `AskUserQuestion`。
3. 宿主等价原生选择工具。
4. Markdown 决策卡：仅作为无原生工具时的降级等待界面，输出后必须停止。
 5. AI 自动选择：仅当用户明确说“你定/直接做/不用问”时可用，状态写 `explicit_auto_permission`。普通“帮我生成/做一组”只授权首轮封面 MVP，不授权批量内页。

原生弹框返回前，不能把推荐项写成用户已选择。Markdown 卡片不能冒充弹框证据。

## 模糊意图补全矩阵

用户输入模糊时，最多用 2-3 个弹框补齐用户真正知道且会改变成品的关键项。需求机会、搜索语义、同类内容和内容竞争由 Skill 自己研究，不能问用户“流量从哪里来”。

| 决策项 | 影响 | 默认选项示例 |
|---|---|---|
| 目标读者 | 标题钩子、正文语气、图片素材 | 新手 / 有经验但卡住的人 / 购买决策者 / 本地客户 |
| 用户素材与不可说边界 | 证据页、产品露出和可信度 | 有过程图 / 只有产品资料 / 可模拟场景 / 不可写的主张 |
| 选题角度 | 封面承诺和内页结构 | 误区纠正 / 真实过程 / 清单方法 / 前后对比 |
| 封面字方向 | 缩略图第一眼 | 反常识 / 结果承诺 / 场景痛点 / 人群筛选 |
| 视觉路线 | 图片风格和素材 | 真实桌面 / 信息卡 / Plog 记录 / 产品细节 / 大字封面 |
| 素材真实度 | 是否可模拟 | 使用用户素材 / 模拟真实素材 / 抽象图解 |
| 平台边界 | 是否可导流、是否像硬广 | 克制承接 / 公开区不导流 / 只在主动问起时回应 |

如果用户已经明确给出某项，就不要再问那项；把它记录为 `user_provided`。

## 弹框 1：方向补全

目的：快速锁定这条笔记到底写给谁、解决什么场景、采用什么角度。

在弹框前，Skill 先输出自己的机会研究摘要：读者的可能需求、需求阶段、可用证据、拟采用的内容目标与理由；用户只需纠正事实或选择方向，不承担市场研究工作。

选项结构：

```json
{
  "title": "先选这条笔记的核心方向",
  "question": "下面哪种方向更接近你要的发布效果？",
  "choices": [
    {
      "id": "mistake-correction",
      "label": "误区纠正：先打破错误认知，再给方法",
      "what_changes": "封面更适合用反常识字句，正文更适合三段式解释",
      "recommended": true
    },
    {
      "id": "real-process",
      "label": "真实过程：像亲历复盘，不像营销稿",
      "what_changes": "图片更偏真实桌面/过程记录，正文更有场景细节",
      "recommended": false
    },
    {
      "id": "checklist",
      "label": "清单方法：直接给可执行步骤",
      "what_changes": "内页会做 6-8 页步骤卡，标题更偏结果承诺",
      "recommended": false
    },
    {
      "id": "manual",
      "label": "我手动补充方向",
      "what_changes": "等待用户输入自定义方向",
      "recommended": false
    }
  ],
  "block_until_response": true
}
```

## 弹框 2：标题/封面字与视觉路线

目的：锁定封面 MVP 的文字和画面方向。

必须让用户能一次性选择或修改：

- 封面主标题。
- 封面副标题或辅助字。
- 主视觉主体。
- 构图：文字区、主体位置、留白。
- 风格：真实、克制、手机随拍、教育信息图、产品细节等。

如果用户选择“按推荐继续”，记录为 `human_confirmed` 的是“按推荐继续”这个选择，而不是模型之后私自扩写的所有细节。扩写部分应标为 `ai_completed_default`。

## 封面 MVP 定义

封面 MVP 同时包含文字和图：

```text
封面文字：主标题 + 副标题 + 不超过 2 个辅助标签。
封面图：主体、场景、构图、文字安全区、风格、画幅。
缩略图验收：手机信息流里先读主标题，主体不遮挡文字，文字不贴边。
批量许可：用户选择“封面通过并批量生成”，或初始请求明确写了“直接全套 / 不用问 / 你定并完整做完 / 直接做完整图文包”。
```

封面 MVP 的确认来源只能是：

- 用户看见封面候选图后明确通过。
- 原生弹框返回“通过并批量生成”。
- 用户原始请求已经明确“你定，不用问，直接做完整图文包”，此时状态是 `explicit_auto_permission`，不是 `human_confirmed`。

普通生成类请求，例如“帮我生成 / 做一组 / 适合发布 / 画面加文字”，只能触发首轮封面 MVP：生成 1 张封面候选并进入验收弹框，不能直接生成 6-8 页内页。

## MVP 确认后的并行生产

封面 MVP 确认后，不再追问范围，直接并行完成：

```text
内容线：发布标题首选 + 至少 8 个标题备选 + 正文 + 标签 + 承接边界。
视觉线：封面最终图 + 6-8 页内页图，每页有独立页面角色和 Image2 Brief。
脚本线：图文页脚本、每页标题、每页唯一要点。
工作台线：完整 15 段工作台写入归档文件。
校验线：图片证据、缩略图可读性、标题承诺兑现、平台边界检查。
交付线：聊天里展示发布版摘要、封面预览、正文、标题备选、标签、图片清单。
```

失败时只问具体失败页的重试或降级，不重新问“是否只要文案”。

## 生成顺序

标准流程：

1. 识别生成类意图，禁止从 HookPrompt scope 降级成纯文案。
2. 原生弹框补齐方向、标题/封面字和视觉路线。
3. 写封面 Image2 Brief。
4. 生成 1 张本轮新封面候选；除非用户要求 A/B，不批量抽卡。
5. 做手机缩略图自检。
6. 若当前宿主有已确认可用的视觉工作台，可把本轮封面放入 3:4 槽位辅助缩略图检查；工作台不可用时直接沿用当前图片预览。
7. 工作台只在本轮 3:4 封面首稿已经可见后按 `visual-workspace-contract.md` 询问一次；不得在机会研究、营销决策或 Brief 阶段抢先问。用户跳过、不可用或启动失败时直接继续。随后用原生弹框让用户选择：`封面通过并批量生成` / `修改文字` / `修改画面` / `先停查看`。工作台中的选中、拖放或槽位 id 不算选择证据。
8. 用户确认批量后，生成完整图文包；若启用工作台，批量结束后建立逐页资产清单，只返修失败页。
9. 输出满意交付形态；完整 15 段工作台仅放归档。

视觉工作台始终是可选增强。Claude Code 没有对应工作台能力时，继续使用 `AskUserQuestion` 和既有图片路线；Codex 没有对应能力时继续使用 `request_user_input` 和既有图片路线。不得因缺少工作台把完整包降级成纯文案。

工作台的同阶段去重、跨阶段新价值门、`explicit_auto_permission` 和 Markdown 降级等待不复用封面 MVP 的确认状态；工作台打开成功也不能把批量生产许可写成 `human_confirmed`。

## 内页生成要求

完整图文包包含封面 + 6-8 页内页。每页内页 Brief 必须包含：

- 页码。
- 页标题。
- 该页唯一要点。
- 画面角色：证据、步骤、对比、清单、故事、结果、收束。
- 主体和场景。
- 构图和文字安全区。
- 与封面风格的连续性。
- 与其他页的差异点，避免重复图。
- 读者在本页完成的认知动作。
- 主体区、主标题区、辅助文字区与禁止遮挡区。
- 为什么本页使用这个构图而不是上一页的构图。

## 不得再次降级的场景

以下场景不得再问“这次做到哪一步”：

- 用户一开始就说要完整图文、封面内页、生成图片或完整发布包。
- 用户已经在前置弹框里确认方向、标题/封面字和视觉路线。
- 用户已经通过封面 MVP，并选择批量生成。
- 用户明确说“你定，直接做”。

触发本 skill 后没有纯文案缩小范围。用户要求暂停时，只能停在当前流程状态并标注 `partial_pending_mvp_confirmation` 或 `blocked_pending_human_decision`；不能改成“只要文案 / 文案+脚本”交付。

## 交付状态

最终状态只能是三类：

```text
complete：封面 + 6-8 内页 + 标题组 + 正文 + 标签 + 承接边界 + 本轮图片证据齐全。
partial：内容完成但图片缺页、图片失败、缺批量许可或只有单张封面。
blocked：缺关键人类决策、缺原生确认、无 Image2/图片路线证据且用户没有同意降级。
```

## 一票否决（MVP 专属）

通用一票否决见 `human-decision-first-workflow.md`「一票否决」（权威 source，本文件不重复）。下面只列 MVP 阶段独有的否决项：

- 封面候选图没生成，就要求用户确认“通过并批量生成”。
- 内页批量生产时每页画面角色相同，像同一张图重复裁切。
- 弹框 2 让用户选标题/封面字时，把模型扩写的细节伪装成 `human_confirmed`（应标 `ai_completed_default`）。
