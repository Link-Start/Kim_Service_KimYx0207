# 人类决策优先流程合同


## 什么时候读

当用户给出模糊意图、要求成品图文、要求出图、要求批量生产、或对图片/排版不满意时，读本文件。

## 核心原则

小红书图文不是靠模型抽卡碰运气。模型只能推荐、解释和执行；关键方向必须由人类决策锁定。

允许继续执行的决策状态只有三种：`human_confirmed`、`explicit_auto_permission`、`blocked_pending_human_decision`。完整定义见 `skill-contract.md`「决策状态合同」（权威 source，本文件不重复定义，避免两处漂移）。下面只讲各状态下如何决策。

`默认假设` 不是继续执行许可。默认假设只能用于展示方案、标题备选、视觉备选和待确认项，不能触发 Image2 生成、批量出图或交付级声明。例外：原始输入已经是生成动作时，记录为 `explicit_auto_permission`，不要写成默认假设。

`缺真实素材` 不是自动阻塞。用户要真实感、真人写真、留学生活、交友、美女风格或模拟真实素材时，可以在 `explicit_auto_permission` 下直接进入 Image2/宿主原生图像；声明素材为生成图即可，不要把“最好实拍/真实授权素材”写成不出图理由。

`文字信息卡` 不是本地 PNG 直通证。用户说图片上文字说明是主体时，仍先走 Image2/宿主原生一体化图文封面；只有文字错误、不可读、排版失败或用户要求可编辑时，才降级到本地排版 PNG。

`已有旧输出` 不是继续执行许可。历史线程、旧 `outputs/` 目录、旧图片文件和示例成稿只能用于对比或参考；除非用户明确要求复用旧包，否则不得把旧图片写成本轮生成结果，也不得用旧文件跳过人类决策。

`生成类意图` 是完整流程许可。用户说“帮我生成小红书”“做图文笔记”“引流贴/种草贴”“封面和内页”“完整发布包”时，默认目标是可发布图文包流程：先锁定方向和封面 MVP，再由封面通过触发批量内页。前置策略、标题、封面字、视觉路线和 Brief 决策完成后，模型应继续生成封面候选；不得再弹出“文案+脚本（推荐）/生成封面/只要文案”这类范围降级问题。

`已完成前置决策` 是执行信号，不是再次询问范围的理由。用户已经在原生弹框里选完方向、标题/封面字和视觉路线后，模型必须进入封面候选生成；不得再问“这次要做到哪一步”。封面验收弹框里若用户选择“通过并批量生成”，模型必须继续生成整套内页；不能停在文案、脚本、单张封面或“下一步再说”。

`优化提示词污染` 必须被忽略。若 HookPrompt 或模型整理出的 `<scope>` 写了“内容形式：小红书图文笔记文案”“不需要制作图片”“只输出可发布文案”，但原始输入只是“用小红书 SKILL 生成/做引流贴/做图文”，不得按纯文案执行。

## 必须有人类决策的节点

1. `策略路线`：写给谁、内容立场、是否商业/课程相关、不能写什么。
2. `选题角度`：主推角度、适合人群、素材要求、风险提醒。
3. `标题与封面字`：标题和封面字必须确认后才能写正文和生成封面。
4. `视觉方案`：真实过程记录、信息图、Plog、产品细节、大字封面等视觉路线。
5. `绘画提示词模板路由`：T1-T10 模板组合必须记录选择理由。
6. `Image2 一体化封面 Brief`：生成前必须锁定文字、构图、主物件、验收标准和 fallback 触发。
7. `封面候选图`：最多生成 1 张主候选；除非用户明确要求 A/B 方案，不得批量抽卡。
8. `封面验收`：手机信息流小图测试通过后，必须立刻调用当前宿主原生选择工具，让用户选“封面通过并批量生成 / 先停查看 / 不通过改 Brief”。用户明确选择批量生成后才进入批量内页；选择先停则等待；选择不通过则改 Brief。只在文字里写“下一步请确认”不算验收。
9. `批量内页生产`：只有封面通过小图测试，且批量生产许可为 `human_confirmed` 或 `explicit_auto_permission` 后才能批量生成。
10. `fallback / 重生成`：失败后必须说明失败类型、修正 Brief，再等待人类确认或明确授权，不得静默重抽。

可选视觉工作台不是新增的人类决策门。它只在封面首稿已经可见、用户能立刻检查 3:4 排版和批注时额外询问一次；跳过、不可用或启动失败后继续第 8 项封面验收。工作台状态、去重和降级规则见 `visual-workspace-contract.md`，不得用工作台打开/选中证据替代本节任何确认。

## 决策弹框选项规则

弹框应服务“方向选择”，不能把用户已经表达的完整目标降级。

允许的前置弹框：

- 选题角度：例如误区清单 / 真实日常 / 过程记录。
- 视觉路线：例如 Plog / 信息卡 / 真实桌面。
- 标题与封面字：选择标题方向和封面主字。

禁止的前置弹框：

- 在用户已要求生成类交付后，问“这次要做到哪一步？”
- 在用户已完成前置选择后，再问“做到哪一步 / 是否只做文案 / 要不要先停在脚本？”
- 把“文案+脚本，不生成图片”标成推荐。
- 引用 HookPrompt 自己添加的“不需要制作图片”作为不出图理由。
- 把“生成封面，不做内页”作为默认停止点。
- 让用户在“完整生成”和“只要文案”之间二次确认。触发本 skill 后没有纯文案路线，只有流程内的暂停、修改或继续批量生成。

需要阶段确认的唯一默认位置是封面验收后：封面通过并批量生成 / 先停查看 / 不通过改 Brief。封面未生成前，不应把“先停在文案”作为推荐路线。

## 聊天与归档的边界

完整 15 段工作台用于归档、校验和回溯；它可以写入 Markdown 文件，也可以在用户要求“完整工作台/调试信息”时展示。最终聊天默认只交付发布版，发布版交付结构权威定义见 `SKILL.md`「聊天最终交付结构」（本文件不重复列举）。

禁止把 `策略补全与决策提醒` 到 `最终成品自检` 15 个段落整段贴进最终聊天，造成用户看不到图片和正文重点。若只有文案、没有实图或没有批量许可，聊天也必须短而清楚地说明 `partial` 或 `blocked`，不要用长工作台稀释阻塞。

## 输出必须写的决策记录

每个完整工作台输出必须包含或可追溯到：

```text
人类决策记录：
- 策略路线：human_confirmed / explicit_auto_permission / blocked_pending_human_decision；来源：...
- 标题与封面字：human_confirmed / explicit_auto_permission / blocked_pending_human_decision；来源：...
- 视觉方案：human_confirmed / explicit_auto_permission / blocked_pending_human_decision；来源：...
- 模板路由与 Brief：human_confirmed / explicit_auto_permission / blocked_pending_human_decision；来源：...
- 封面候选图：human_confirmed / explicit_auto_permission / pending_user_review / blocked_pending_human_decision；来源：...
- 批量生产许可：human_confirmed / explicit_auto_permission / blocked_pending_human_decision；来源：...
- 继续执行许可：yes / no；原因：...
- 原生确认证据：request_user_input / AskUserQuestion / 阻塞：缺少原生确认工具；来源：...
```

可以放在 `策略补全与决策提醒`、`图片生成与配图提示词`、`封面视觉验证` 和 `批量图片生产` 中，但不能完全缺失。

## 抽卡防线

- 不得为了“更好看”连续生成多张随机封面让用户挑。
- 如果用户要求多个方向，必须先让用户在方向层选择，例如真实桌面 / 信息卡 / 大字海报，而不是同一 Brief 多次随机出图。
- 每次重新生成必须有可描述的修改：标题位置、主体占比、文字层级、光线、真实素材、禁止项，而不是只换 seed。
- AI 自检可以判定不通过，但不能自己无限重跑；最多提出修正 Brief 和下一步确认。
- 批量内页不允许在封面未通过小图测试时提前生成；封面通过后必须弹框让用户选择批量生成、先停查看或改 Brief。若用户在弹框里确认批量生成，不得只停在 1 张封面。
- 看到同题旧输出时不得当作本轮封面候选或批量内页；旧输出只可作为“历史参考”，本轮仍要从原生选择、封面生成和批量许可重新取证。
- 生成类请求的前置方向已经确认后，下一步直接生成封面候选；不要新增“是否做到文案/封面/全文”的范围确认。

## 通过标准

- 用户能看到哪些是已确认、哪些是默认假设、哪些仍阻塞。
- 模型推荐和人类决策分开写，不能把推荐默认伪装成用户选择。
- Image2 生成前已经有明确模板路由和 Brief。
- 失败后进入修正 Brief，而不是继续抽卡。
- 交付状态和人类确认状态一致：缺少批量生产许可或缺少批量图证据时只能是 `partial` 或 `blocked`。

## 一票否决

- 写“推荐方案”后直接继续生成，未记录人类确认或明确自动授权。
- 写“你选了/已确认”，但线程里没有用户原话、`request_user_input` 或 `AskUserQuestion` 证据。
- 用户只说“不要太 AI 味”，模型自己确定标题、封面、模板、生成图并称完整交付。
- 只因缺真实素材就停止所有图片生产，没有按用户要求进入模拟真实素材生成。
- 封面候选图不满意后，直接再生成多张，没有先改 Brief。
- 手机信息流测试未通过就批量出图。
- 用户要求完整视觉交付且已经给出完整交付许可时，只生成 1 张封面就结束。
- 普通 Markdown 选项卡冒充 Codex `request_user_input` 或 Claude Code `AskUserQuestion`。
- 封面生成后只在文字里写“下一步确认/通过再批量”，没有真实原生弹框证据。
- 用户已经要求图文/生成/引流贴且前置决策已完成，仍把“文案+脚本，不生成图片”设为推荐路线。
- 用户已经通过封面验收弹框选择批量生成，仍只输出文案/脚本/单张封面，或再次询问要不要生成图片。
- 把完整 15 段工作台作为最终聊天输出，导致聊天里堆满过程信息，缺少满意版本那种图片预览、标题、正文和标签的发布版交付。
- 复用历史 `outputs/` 或旧生成图，却写成本轮已经生成图片。
